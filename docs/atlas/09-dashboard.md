# Atlas — Dashboard App

> Stage 8 deliverable. `apps/dashboard`: the Next.js App Router app hosting the
> visual page editor. SSR-verified headlessly (full editor surface renders with
> drop zones, palette, canvas, inspector).

## Structure

```
apps/dashboard/
  next.config.ts            styledComponents compiler + transpilePackages (@atlas/*)
  src/app/layout.tsx        styled-components SSR registry + Shell + structural.css
  src/app/page.tsx          pages index (per-tenant)
  src/app/editor/[pageId]/  editor route (server loads draft → client Editor)
  src/app/editor/actions.ts Server Actions: save / publish / schedule (Zod-validated)
  src/lib/cms-stub.ts       in-memory persistence seam — the @atlas/cms contract
  src/components/shell/     AuthGuard placeholder + tenant switcher + chrome
  src/components/editor/    Editor, StructureTree, Inspector, styles
```

## How the editor composes the platform

- **State** — `usePageEditor` (`@atlas/page-builder`): all editing semantics live in the
  verified headless core. The dashboard adds chrome only.
- **Canvas** — renders the tenant's *actual* site output: the real `PageRenderer` under a
  `TenantThemeProvider`, with each component wrapped in a selectable frame
  (`data-node-id`). Editing overlays wrap output; they never replace rendering.
- **Drop zones** — the structure panel's region targets come from `resolveRegions()`
  metadata; drops are validated by `canDrop` (accepts/maxChildren/descendant rules) with
  the verdict enforced at drop time. HTML5 drag carries `{kind, paletteId|nodeId}`.
- **Palette** — `buildPalette()` groups from both registries; new registrations appear
  with zero editor changes (ADR-0005).
- **Inspector** — generated by `deriveEditableFields` for components *and* layout
  options (same schema→fields path, ADR-0006). Inline-flagged fields group under
  "Content"; duplicate/remove act on the selection.
- **Publish workflow** — toolbar → Server Actions (`saveDraftAction` validates against
  `pageDefinitionSchema`, `publishAction`, `scheduleAction` with future-time guard) →
  cms layer. `dirty`/`mark-saved` drive the save state line.
- **Responsive previews** — canvas sheet width toggles (375/768/1100) via CSS var.

## The persistence seam

`cms-stub.ts` is deliberately shaped as the `@atlas/cms` contract
(`listPages/getPageDraft/savePageDraft/publishPage/schedulePage`), validating at the
boundary exactly as the Supabase implementation will (RLS + `page_versions`
append-only, Stage 2). Swapping stub → Supabase changes no editor code. Auth is an
explicit `AuthGuard` boundary placeholder wired for `@atlas/authentication` (Stage 10);
the dashboard chrome renders under the *platform* theme — tenant themes apply only
inside the canvas.

## Known v1 scopes

- Inline editing is surfaced through the inspector's Content group; true on-page
  contenteditable overlays (anchored to `data-node-id`) are the planned enhancement.
- Tenant switcher lists the demo tenant until cms memberships land.
- `next build` of the app awaits the repo's in-flight bun→pnpm toolchain completion.
